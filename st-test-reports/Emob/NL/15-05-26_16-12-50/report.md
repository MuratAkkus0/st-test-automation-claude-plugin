### Captured inline conversion-tag snippet

```js
var st = document.createElement("script");
st.setAttribute("src", "https://www.moebel.de/partner/push.js");        // ← WRONG: German script
st.setAttribute("data-partner-key", "609d0e02-16de-4f17-abcb-190dad1d8701");
st.setAttribute("onload", "sendSale()");
document.head.appendChild(st);
function sendSale() {
  MOEBEL_SALES.sale({
    total:    google_tag_manager["rm"]["48470055"](20),
    shipping: google_tag_manager["rm"]["48470055"](21),
    currency: "EUR",
    orderId:  "5000084845",
    items:    google_tag_manager["rm"]["48470055"](22)
  });
}
```

The script URL determines which market endpoint the sale lands on:
- `www.moebel.de/partner/push.js`  → sends to `redirect.moebel.de/.../de/sales`
- `www.meubelo.nl/partner/push.js` → sends to `redirect.meubelo.nl/.../nl/sales`

Emob is currently loading the German script on the Dutch storefront, so every Dutch sale is recorded as a German conversion in our backend.

### Verdict

❌ **FAIL — wrong market endpoint.** The Base Tag works correctly and the conversion call is structurally sound (payload values resolved, order id matches), but the call is being routed to the German tracking endpoint instead of the Dutch one. The partner has copy-pasted their German GTM Conversion-Tag onto their Dutch storefront without swapping the `push.js` source URL.

## Root cause analysis

The Sales Tracking Base Tag and Conversion Tag are both technically firing, with correctly resolved payload values. The single point of failure is the `push.js` script source URL on the order-confirmation page:

- Emob's Conversion Tag on `emob.nl` (NL storefront) injects `https://www.moebel.de/partner/push.js`.
- That script's `MOEBEL_SALES.sale(...)` implementation posts to `https://redirect.moebel.de/api/1.0/moebel/de/sales` — the **German** sales-ingestion endpoint.
- The Dutch storefront must instead load `https://www.meubelo.nl/partner/push.js`, whose `sale(...)` posts to `https://redirect.meubelo.nl/api/1.0/moebel/nl/sales` — the **Dutch** sales-ingestion endpoint.

Because the German script is loaded, all conversions placed on `emob.nl` are silently registered as conversions on the German Emob integration in our ACM tool. None of them reach the Dutch market account. The PARTNER_KEY visible in the URL (`609d0e02-16de-4f17-abcb-190dad1d8701`) is whatever key the partner assigned to the German integration — this needs to be cross-checked against ACM, and the dedicated NL PARTNER_KEY (if a different one was issued for NL) must be plugged in alongside the corrected script URL.

This is a textbook "integration copied between markets without swapping market-specific resources" failure mode — same family as the documented "Wrong PARTNER_KEY across markets" scenario, but in this case both the script source AND potentially the key need to be checked.

---

## Recommendations

1. **Replace the `push.js` source** in the GTM Conversion Tag on the Dutch storefront from `https://www.moebel.de/partner/push.js` to `https://www.meubelo.nl/partner/push.js`. Publish the GTM container.
2. **Cross-check the PARTNER_KEY** `609d0e02-16de-4f17-abcb-190dad1d8701` against ACM. If a dedicated NL PARTNER_KEY exists, replace the `data-partner-key` attribute with that value at the same time as step 1.
3. **Fix the data-layer reference errors** (`ecommerce is not defined`, `tp is not defined`) — these are GTM container issues that do not block sales tracking today but suggest other tags on the success page are broken.
4. **Re-test after the fix** by repeating the click-through from `meubelo.nl/shops/emob` and placing one order. The expected outcome is a fetch to `https://redirect.meubelo.nl/api/1.0/moebel/nl/sales?key=<NL_KEY>` with the same payload structure.
